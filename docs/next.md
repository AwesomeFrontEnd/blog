# 服务器端组件

服务器和客户端组件允许开发者跨服务器和客户端构建应用，组合客户端侧的应用用丰富的交互性使用过币传统服务器渲染更强的性能。

## 服务器端组件

和考虑 React 如何改变了我们构建用户界面的思考方式，React服务器端组件介绍了一种新的心智模型，来构建混合应用，这些应用在服务器端和客户端都能运行。

取代了React渲染在客户端侧渲染你的整个应用（比如单页应用的案例），React 现在给你灵活性，去选择哪里渲染你的组件，基于他们的目的。

如果我们要把这个页面分离进更小的组件们，你会注意到主药的组件是非交互性的，能作为服务器端组件被渲染在服务器上。对于那些更小颗粒度的交互性UI组件，我们可以放进客户端组件里。这，就是Next.js 服务器端组件的最初的目的。

## 为什么是服务器端组件？

所以，你可能会想，为什么是服务器端组件？使用它们，和使用客户端组件有说明优势？

服务器端组件允许开发者更好地利用服务器的基础设施。比如，你能从服务器获取数据，距离你的数据库更接近，并且保持大的依赖过去会影响服务器端JavaScript在服务器上的打包大小，从而影响性能。服务器端会让写一个React应用的感觉更像是PHP或者 Ruby on Rails,但有了React的强大的能力和灵活性，和把用户界面进行组件的模板化。

使用服务器端组件，首页加载更快了，客户端侧JavaScript的打包速度减少了，基础的客户端侧运行时是 **可缓存的** 和 **可预测的** 在大小上，但不会增加，当你的应用变的健壮之后。增加的JavaScript 会作为客户端侧的交互性被在你的整个客户端组件中使用。

当一个路由使用 Next.js 加载，最初的HTML在服务器端渲染，这个HTML接下来会被逐步在浏览器中增强，运行客户端结果应用，并且增加交互性，同时通过加载Next.js和React客户端侧的运行时。

为了让往客户端侧的组件转换更容易，所有的在App 路由里的组件默认是客户端组件，包括特殊的文件，和托管的组件。这让你自动采纳他们不用多余的操作，能极大的提高性能。当然你也可以可选的使用“use client”指令来对他们使用客户端组件。
